# Configuration Guide

## Environment Variables

Set in `docker-compose.yml` under `environment`:

| Variable | Required | Default | Description |
|----------|----------|---------|-------------|
| `OWNER_PUBKEY` | ✅ | — | Hex pubkey of the instance owner. Determines who gets Owner role in the UI. |
| `NEO4J_PASSWORD` | ✅ | — | Password for Neo4j database. Used by the Express API. |
| `DOMAIN_NAME` | No | `localhost` | Domain name for the instance. Used in nginx config. |

## Two-Layer Settings System

Tapestry uses a two-layer configuration pattern:

```
defaults.json (shipped with code, read-only)
     +
settings.json (user overrides, persistent volume)
     =
merged config (used at runtime)
```

### Layer 1: Defaults

**File:** `src/concept-graph/parameters/defaults.json`

This file ships with the code and is tracked in git. It contains sensible defaults for all settings. **Do not edit this file directly** — your changes will be overwritten on updates.

### Layer 2: User Overrides

**File:** `/var/lib/brainstorm/settings.json` (inside Docker)

This file lives on the persistent `tapestry-data` volume. It's created automatically when you change settings through the UI. It contains **only the values you've changed** — everything else falls through to defaults.

### How merging works

At runtime, `getSettings()` deep-merges the two layers. For example:

```json
// defaults.json
{
  "aRelays": {
    "aProfileRelays": ["wss://purplepag.es", "wss://profiles.nostr1.com"],
    "aDListRelays": ["wss://dcosl.brainstorm.world"]
  }
}

// settings.json (user override)
{
  "aRelays": {
    "aProfileRelays": ["wss://purplepag.es", "wss://profiles.nostr1.com", "wss://relay.damus.io"]
  }
}

// merged result
{
  "aRelays": {
    "aProfileRelays": ["wss://purplepag.es", "wss://profiles.nostr1.com", "wss://relay.damus.io"],
    "aDListRelays": ["wss://dcosl.brainstorm.world"]
  }
}
```

Arrays are **replaced** (not concatenated). Objects are **deep-merged**.

## Settings Reference

### Relays (`aRelays`)

All relay lists are arrays of WebSocket URLs (`wss://` or `ws://`). Changes take effect **immediately** — no restart needed.

| Key | Default | Kinds | Purpose |
|-----|---------|-------|---------|
| `aProfileRelays` | `purplepag.es`, `profiles.nostr1.com` | 0 | Fetch user profiles (name, avatar, etc.) |
| `aPopularGeneralPurposeRelays` | `relay.damus.io`, `relay.primal.net`, `nos.lol`, `relay.nostr.band` | Various | General-purpose relays for broad reach |
| `aDListRelays` | `dcosl.brainstorm.world` | 9998, 9999, 39998, 39999 | DList events (headers and items) |
| `aWotRelays` | `wot.grapevine.network` | 3, 1984, 1000 | Web of trust signals (follows, reports, mutes) |
| `aTrustedAssertionRelays` | `nip85.nostr.band`, `nip85.brainstorm.world`, `nip85.nostr1.com` | 30382–30385 | NIP-85 trusted assertions |
| `aTrustedListRelays` | `nip85.brainstorm.world`, `dcosl.brainstorm.world` | 30392–30396 | NIP-85 trusted lists |
| `safeModeRelays` | `nip85.brainstorm.world`, `dcosl.brainstorm.world`, `wot.grapevine.network` | Various | Minimal relay set for safe mode |
| `aOutboxRelays` | *(empty)* | Various | Outbox model publishing |

### Concept UUIDs (`conceptUUIDs`)

These identify the canonical DList headers that define the knowledge graph schema. They're in `a-tag` format: `kind:pubkey:d-tag`. Changes **require a restart**.

| Key | Description |
|-----|-------------|
| `relationship` | The "Relationships" concept — parent of all relationship type definitions |
| `relationshipType` | The "Relationship Types" concept |
| `nodeType` | The "Node Types" concept |
| `set` | The "Sets" concept |
| `superset` | The "Supersets" concept — the canonical superset of all supersets |
| `JSONSchema` | The "JSON Schemas" concept |
| `property` | The "Properties" concept |

### Relationship Type UUIDs (`relationshipTypeUUIDs`)

These identify the canonical DList items that define relationship types. Changes **require a restart**.

| Key | Description |
|-----|-------------|
| `CLASS_THREAD_INITIATION` | ListHeader → Superset |
| `CLASS_THREAD_PROPAGATION` | Superset → Superset |
| `CLASS_THREAD_TERMINATION` | Superset → Element |
| `IS_A_PROPERTY_OF` | Property → JSONSchema |
| `IS_THE_JSON_SCHEMA_FOR` | JSONSchema → ListHeader |
| `ENUMERATES` | Concept → Property |

## Settings API

All endpoints require **Owner** authentication (NIP-07 login).

| Method | Endpoint | Description |
|--------|----------|-------------|
| `GET` | `/api/settings` | Returns merged settings, defaults, and overrides |
| `GET` | `/api/settings/defaults` | Returns only shipped defaults |
| `GET` | `/api/settings/overrides` | Returns only user overrides |
| `PUT` | `/api/settings` | Update overrides (deep merge with existing). Body: JSON object. |
| `DELETE` | `/api/settings/:keyPath` | Reset a specific key to default. e.g., `DELETE /api/settings/aRelays.aProfileRelays` |

The `PUT` response includes a `needsRestart` flag when the change affects concept UUIDs or system parameters.

## brainstorm.conf

The main server config file at `/etc/brainstorm.conf` inside Docker. This is generated at container build time and contains:

- File paths for all services
- Neo4j connection details
- Relay configurations (legacy format, being migrated to settings system)
- Owner pubkey
- Tapestry Assistant keypair (`BRAINSTORM_RELAY_PRIVKEY`, `BRAINSTORM_RELAY_PUBKEY`)
- Session secret

> **Note:** `brainstorm.conf` is the legacy configuration format. The two-layer settings system (`defaults.json` + `settings.json`) is the new standard for user-facing configuration. Over time, more settings will migrate from `brainstorm.conf` to the settings system.
